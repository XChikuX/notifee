## 1. Install from npmjs.com

Install Notifee to the root of your React Native project with [npm](https://www.npmjs.com/) or
[Bun](https://bun.sh/)

```bash
# Using npm
npm install --save @psync/notifee

# Using Bun
bun add @psync/notifee
```

## 2. Configuration Changes

### 2a. Android API versions

The values of `compileSdkVersion` and `targetSdkVersion` might need to be changed. Notifee is built and tested against Android API 36, so use `compileSdkVersion` 36 or higher. `targetSdkVersion` 35 or higher is recommended, and if your app is going to request permission on Android 13 via `requestPermission`, `targetSdkVersion` must be at least 33. These settings are in `/android/build.gradle`.

```gradle
buildscript {
  ext {
    compileSdk = 36
    targetSdk = 35
    minSdk = 28
    ...
  }
  ...
}
```

Your Android project can fail to build with missing symbols if these settings are too low, and newer notification behaviors are only validated against current Android SDK levels.

### 2b. Build tool versions

| Tool                        | Minimum Version |
| --------------------------- | --------------- |
| Android Gradle Plugin (AGP) | 8.8.2           |
| Gradle                      | 8.13.x          |
| Kotlin                      | 2.0.21          |
| Java (JDK)                  | 17 or 21        |
| NDK                         | 27.1.12297006+  |
| Build Tools                 | 36.0.0          |

Your android project will fail to build with errors related to resource compilation / packaging if you do not use a new enough android gradle plugin / gradle combination.

## 3. Autolinking with React Native

Users on React Native 0.60+ automatically have access to "[autolinking](https://github.com/react-native-community/cli/blob/master/docs/autolinking.md)",
requiring no further manual installation steps. To automatically link the package, rebuild your project:

```bash
# For iOS
cd ios/ && pod install --repo-update
npx react-native run-ios

# For Android
npx react-native run-android
```

## Miscellaneous

### Expo Support

Notifee now ships an official Expo config plugin for `expo prebuild`. For Expo projects, it helps to think about setup in two parts:

- Use `expo-build-properties` for app-wide native build settings such as Android SDK levels and the main iOS deployment target.
- Use `@psync/notifee` for notification-specific native changes such as APNs entitlements, iOS background modes, Notification Service Extension setup, Android notification icons, Android notification sounds, and iOS notification sound bundling.

Install `expo-build-properties` first:

`npx expo install expo-build-properties`

Then add both plugins to your Expo config:

```js
{
  "expo": {
    "plugins": [
      [
        "expo-build-properties",
        {
          "android": {
            "compileSdkVersion": 36,
            "targetSdkVersion": 36,
            "buildToolsVersion": "36.0.0"
          },
          "ios": {
            "deploymentTarget": "15.1"
          }
        }
      ],
      [
        "@psync/notifee",
        {
          "apsEnvMode": "development",
          "backgroundModes": ["remote-notification"],
          "androidIcons": [
            {
              "name": "ic_stat_notify",
              "path": "./assets/notifications/ic_stat_notify.png",
              "type": "small"
            }
          ],
          "androidSoundFiles": [
            {
              "name": "message_chime",
              "path": "./assets/notifications/message_chime.mp3"
            }
          ],
          "iosSoundFiles": ["./assets/notifications/chime.wav"],
          "notificationServiceExtension": {
            "enabled": true
          }
        }
      ]
    ]
  }
}
```

**Supported plugin options:**

| Option                             | Type                        | Description                                                                                                                  |
| ---------------------------------- | --------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| `apsEnvMode`                       | `string`                    | `"development"` or `"production"`. Usually unnecessary if `expo-notifications` plugin sets this.                             |
| `backgroundModes`                  | `string[]`                  | Opt-in. The plugin does not add `remote-notification` unless set.                                                            |
| `enableCommunicationNotifications` | `boolean`                   | Enables iOS communication-notification capability.                                                                           |
| `androidIcons`                     | `Array<{name, path, type}>` | Android notification icon resources. `type` is `"small"` or `"large"`.                                                       |
| `androidSoundFiles`                | `Array<{name, path}>`       | Copies sound files into `android/app/src/main/res/raw`.                                                                      |
| `iosSoundFiles`                    | `string[]`                  | Copies iOS notification sounds (`.wav`, `.aif`, `.aiff`, `.caf`) into both app and extension targets.                        |
| `notificationServiceExtension`     | `object`                    | `{ enabled?, name?, bundleIdentifier?, deploymentTarget?, appGroupName?, customSourceFilePath?, entitlements?, infoPlist? }` |
| `appleDevTeamId`                   | `string`                    | Usually unnecessary if `expo.ios.appleTeamId` is already set.                                                                |

Notes:

- `androidSoundFiles` copies local files into `android/app/src/main/res/raw`. At runtime, reference the configured resource name such as `sound: "message_chime"` on your channel or notification.
- Android accepts a broader range of notification sound containers than iOS. In practice, `.mp3`, `.wav`, and `.ogg` are the most predictable Android choices, while iOS notification sounds must stay `.wav`, `.aif`, `.aiff`, or `.caf`.
- `notificationServiceExtension.deploymentTarget` in the Notifee plugin only changes the generated Notification Service Extension target. Set the main app deployment target with `expo-build-properties` or your native project settings.
- If `expo.ios.appleTeamId` is already present in your Expo config, you usually do not need to also set `appleDevTeamId` in the Notifee plugin options.
- When `iosSoundFiles` and `notificationServiceExtension.enabled` are both enabled, the plugin wires those sounds into both the main app target and the extension during the same prebuild pass.
- Small icons should be transparent, monochrome status-bar assets (PNG). The plugin warns when a small icon source is not a PNG or when an icon source is not square.
- Flat legacy aliases remain supported for backward compatibility: `enableNotificationServiceExtension`, `notificationServiceExtensionName`, `notificationServiceExtensionBundleIdentifier`, `iosDeploymentTarget`, `customNotificationServiceFilePath`, `appGroupName`, `notificationServiceExtensionEntitlements`, and `notificationServiceExtensionInfoPlist`.

Finally, ensure you run `npx expo prebuild` and rebuild your app as described in [Add custom native code](https://docs.expo.dev/workflow/customizing/) and [Continuous Native Generation](https://docs.expo.dev/workflow/prebuild/).

If you are building with EAS, you usually do not need to pin a custom Android image. Current EAS Build images already use Java 17 by default when you rely on the normal `auto` image selection. If you choose to pin `image` manually, use a current Java 17 image from Expo's [build server infrastructure reference](https://docs.expo.dev/build-reference/infrastructure/).
